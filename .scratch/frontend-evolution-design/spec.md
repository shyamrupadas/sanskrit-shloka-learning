# Evolution Design small для фронтенда

Status: ready-for-human
Accepted: 2026-07-08

## Problem Statement

Архитектурное решение использовать Evolution Design для фронтенда зафиксировано, но код ему не соответствует. Композиция приложения, страницы, глобальная session/API-логика, UI-примитивы и тесты организованы по несогласованным признакам. Архитектурные направления зависимостей и public API модулей не проверяются автоматически.

Из-за этого разработчику или агенту приходится заново определять место для каждого изменения, глобальная логика смешивается с пользовательскими сценариями, а крупные модули продолжают расти без явных интерфейсов. Монолитные admin- и test-модули уже затрудняют локальные изменения и повышают риск регрессий.

Пользователь не видит архитектуру напрямую, но зависит от ее последствий: существующие сценарии должны оставаться стабильными, а новые возможности заучивания и повторения шлок должны добавляться без замедления и случайных поломок соседних экранов.

## Solution

Привести frontend к Evolution Design small с тремя верхнеуровневыми слоями: `app`, `features`, `shared`.

`app` будет владеть запуском, глобальными providers, маршрутизацией, guards и layouts. Основная функциональность будет организована в крупные самостоятельные фичи `auth`, `dashboard`, `library`, `settings`, `admin`. Глобальная session/API-инфраструктура, i18n, UI kit и общие тестовые средства будут принадлежать `shared`.

Архитектурные зависимости и public API фич будут проверяться ESLint. Текущие правила архитектуры находятся в `docs/architecture/frontend.md`: во время миграции документ также задает временную legacy-политику. Scoped agent instructions направляют агента к этому документу и проверкам, а workflow архитектурного аудита и рефакторинга находится в project skill.

Рефакторинг выполняется пошагово с сохранением существующего поведения, маршрутов, API-контракта и визуального дизайна. Каждый шаг оставляет frontend в проверяемом рабочем состоянии.

Локальный репозиторий `sadhana` используется как близкий образец формы ED small: `app` для композиции, route-level `*.page.tsx` как допустимые public entry points фич, `shared/model` для route constants и session, а также декларативная проверка границ через ESLint. Версии зависимостей и прикладные решения из `sadhana` не копируются автоматически.

## User Stories

1. Как Пользователь, я хочу продолжать входить в Учетную запись после рефакторинга, чтобы архитектурные изменения не нарушили доступ к моему прогрессу.
2. Как Пользователь, я хочу продолжать регистрировать Учетную запись с прежней валидацией, чтобы поведение формы не изменилось.
3. Как Пользователь, я хочу видеть прежнюю навигацию и маршруты, чтобы рефакторинг не изменил привычный способ работы с приложением.
4. Как Пользователь, я хочу продолжать открывать дашборд, Общую библиотеку шлок и настройки, чтобы существующие сценарии оставались стабильными.
5. Как Пользователь, я хочу продолжать находить Шлоки и менять их Статус шлоки пользователя, чтобы архитектурная работа не повлияла на библиотеку.
6. Как Администратор, я хочу продолжать использовать Админку для управления Источниками шлоки и Шлоками, чтобы рефакторинг не нарушил наполнение Общей библиотеки шлок.
7. Как владелец продукта, я хочу отделить архитектурный рефакторинг от новой функциональности, чтобы результат можно было проверить независимо.
8. Как владелец продукта, я хочу отложить рефакторинг дизайн-системы до завершения архитектурной работы, чтобы не смешивать структурные и визуальные изменения.
9. Как frontend-разработчик, я хочу видеть только `app`, `features` и `shared` на верхнем уровне исходного кода, чтобы место ответственности определялось однозначно.
10. Как frontend-разработчик, я хочу хранить route-level страницы внутри соответствующих фич, чтобы пользовательский сценарий был локализован в одном модуле.
11. Как frontend-разработчик, я хочу хранить глобальную композицию приложения в `app`, чтобы фичи не зависели от часто меняющейся orchestration-логики.
12. Как frontend-разработчик, я хочу хранить глобальную session-модель отдельно от `auth`, чтобы защищенные фичи не зависели от реализации входа и регистрации.
13. Как frontend-разработчик, я хочу использовать единый модуль API-инфраструктуры, чтобы создание клиента и обработка ошибок не дублировались между фичами.
14. Как frontend-разработчик, я хочу хранить только действительно глобальные UI-примитивы и инфраструктуру в `shared`, чтобы этот слой не превратился в склад случайного кода.
15. Как frontend-разработчик, я хочу начинать новые модули с простой структуры, чтобы ED не создавал преждевременную папочную церемонию.
16. Как frontend-разработчик, я хочу усложнять отдельный модуль только после появления фактической сложности, чтобы структура эволюционировала вместе с кодом.
17. Как frontend-разработчик, я хочу использовать public API многофайловых фич, чтобы вызывающий код не зависел от их внутреннего устройства.
18. Как frontend-разработчик, я хочу иметь возможность оставить простой `dashboard` однофайловым модулем, чтобы не создавать shallow-модули без пользы.
19. Как frontend-разработчик, я хочу структурировать `auth` по model- и UI-ответственностям, чтобы формы и submit-логика имели понятные интерфейсы.
20. Как frontend-разработчик, я хочу структурировать `library` по отображению, API state и чистой логике поиска/действий, чтобы изменения библиотеки оставались локальными.
21. Как frontend-разработчик, я хочу оставить `admin` одной крупной фичей с внутренними подмодулями, чтобы связанные административные сценарии не превратились в искусственно связанные верхнеуровневые фичи.
22. Как frontend-разработчик, я хочу переиспользовать внутренние модели create/edit форм в `admin`, чтобы нормализация и validation не дублировались.
23. Как frontend-разработчик, я хочу создавать будущие фичи заучивания и повторения только вместе с функциональностью, чтобы в коде не появлялись пустые заготовки.
24. Как сопровождающий проекта, я хочу автоматически запрещать зависимости `shared` от `features` и `app`, чтобы нижний слой оставался независимым.
25. Как сопровождающий проекта, я хочу автоматически запрещать зависимости `features` от `app`, чтобы фичи можно было понимать и тестировать независимо от композиции.
26. Как сопровождающий проекта, я хочу автоматически обнаруживать обход public API фич, чтобы внутренние рефакторинги не ломали внешних потребителей.
27. Как сопровождающий проекта, я хочу не создавать слой `services` до доказанной необходимости, чтобы проект оставался в согласованной модификации ED small.
28. Как сопровождающий проекта, я хочу удалить legacy-структуру и временные compatibility imports после миграции, чтобы в проекте не осталось двух архитектур одновременно.
29. Как агент, выполняющий frontend-задачу, я хочу получать короткую инструкцию о том, когда читать живую спецификацию архитектуры, чтобы использовать актуальные правила без их дублирования.
30. Как агент, выполняющий архитектурный рефакторинг, я хочу использовать project skill с ED-workflow, чтобы подробная методика загружалась только для архитектурных задач.
31. Как reviewer, я хочу получать небольшие последовательные изменения с зеленым lint, typecheck и тестами, чтобы проверять рефакторинг поэтапно.
32. Как reviewer, я хочу видеть в каждом модуле небольшой внешний интерфейс и скрытую реализацию, чтобы оценивать зависимости без чтения всех внутренних файлов.
33. Как тестировщик, я хочу проверять поведение фич через их публичные интерфейсы, чтобы тесты переживали внутренние перемещения файлов.
34. Как тестировщик, я хочу проверять providers, маршрутизацию, guards и layouts отдельно от поведения фич, чтобы app-тесты описывали только композицию.
35. Как тестировщик, я хочу использовать общий test harness и существующий fetch-based API adapter, чтобы feature-тесты не дублировали настройку окружения.
36. Как тестировщик, я хочу сохранить E2E-проверки критических пользовательских маршрутов, чтобы подтвердить неизменность поведения всего приложения.
37. Как frontend-разработчик, я хочу иметь возможность использовать route-level `*.page.tsx` как public entry point фичи, чтобы простые маршруты не требовали лишнего barrel-файла.
38. Как сопровождающий проекта, я хочу использовать поддерживаемый ESLint-инструмент для архитектурных границ, чтобы правила ED small были декларативными и понятными reviewer'ам.
39. Как сопровождающий проекта, я хочу видеть React hooks ошибки в lint, чтобы архитектурные переносы не создавали скрытых проблем жизненного цикла.
40. Как reviewer, я хочу запускать frontend lint без лишнего build dependency, чтобы проверка архитектурных правил была быстрой и не смешивалась с генерацией/сборкой.

## Implementation Decisions

- Использовать ED small как текущую модификацию архитектуры. Верхнеуровневые слои — `app`, `features`, `shared`; отдельные слои `pages` и `services` не используются.
- `app` владеет React bootstrap, глобальными providers, TanStack Router, route guards, authenticated/admin layouts и композицией фич.
- `features` содержит крупные пользовательские возможности: `auth`, `dashboard`, `library`, `settings`, `admin`.
- `admin` остается одной фичей с внутренними подмодулями `catalog`, `source-editor`, `shloka-editor`.
- Будущие `learning`, `review` и другие фичи не создаются до реализации соответствующей функциональности.
- `shared` содержит глобальную API-инфраструктуру, session-модель, i18n, route-константы, UI kit, общие utilities и тестовую инфраструктуру.
- UI kit переносится без изменения визуального дизайна. Новые переиспользуемые визуальные компоненты будут определены в отдельном рефакторинге дизайн-системы.
- Session-модуль скрывает local storage, восстановление и очистку сессии, создание авторизованного API-клиента и logout. `auth` отвечает только за пользовательские сценарии входа и регистрации.
- Не вводить отдельный port или adapter без двух реальных реализаций. Существующий injectable fetch generated API client уже является достаточным seam для production и test adapters.
- Страницы принадлежат фичам. Если экран компонует несколько независимых фич, композиция принадлежит `app`.
- Простые фичи могут оставаться single-file modules. Flat/grouped структура, группы `ui`, `model`, `lib`, `api` и подмодули вводятся по фактической сложности.
- Многофайловые фичи предоставляют небольшой public API. Допустимые внешние entry points фичи — `index.ts(x)` и route-level `*.page.tsx`; внутренние hooks, формы, formatters и model helpers не экспортируются.
- Прямые зависимости между фичами допустимы в ED small, но в согласованном целевом состоянии не требуются. Если такая зависимость появится, она проходит через public API импортируемой фичи.
- ESLint автоматически запрещает `shared → features/app`, `features → app` и импорт внутренних файлов фич в обход public API. Для архитектурных границ предпочтителен поддерживаемый декларативный plugin; кастомное правило допустимо только как fallback при несовместимости.
- ESLint также проверяет React hooks. Fast-refresh правило может работать warning-level, чтобы подсвечивать рискованные exports без блокировки shadcn-style UI kit.
- Frontend lint запускается из общего project workflow и не должен зависеть от build/generate tasks, если lint не читает их результаты.
- `docs/architecture/frontend.md` является источником текущих архитектурных правил. ADR фиксирует причины решения, ESLint исполняет проверяемую часть, scoped agent instructions маршрутизируют к документу и проверкам, а project skill описывает workflow аудита и изменения архитектуры.
- Пока legacy-код существует, scoped agent instructions требуют читать живую спецификацию перед любой frontend-задачей. После финального удаления legacy обычная локальная правка использует короткие инструкции и lint, а спецификация читается при изменении файлов, зависимостей, interfaces или ответственности модулей.
- Project skill не дублирует живую спецификацию. Во время архитектурной серии он содержит безопасный migration workflow; после финальной задачи legacy-specific инструкции удаляются, а skill остается для аудита и эволюции ED small.
- Миграция выполняется снизу вверх: исполняемые ограничения, `shared`, session/API, `app`, фичи, финальное удаление legacy-структуры.
- Route constants принадлежат `shared` и вводятся там, где они не ухудшают типизацию TanStack Router.
- Временные compatibility imports допустимы только внутри незавершенного миграционного шага и не остаются в целевом состоянии.
- Каждый шаг сохраняет route paths, route params, query keys, localStorage keys, API-вызовы и пользовательское поведение.
- Public HTTP/API-контракт, generated client и backend не изменяются.
- Схема базы данных и данные не изменяются; миграции базы данных не требуются.
- Каждая реализационная задача должна быть независимо выполнима агентом и завершаться зеленым lint, typecheck и unit test. Финальная задача дополнительно запускает build и E2E.
- Целевое состояние не содержит одновременно legacy- и ED-структуру.
- В финальной задаче ESLint начинает запрещать неизвестные frontend-файлы и зависимости; временные исключения и migration mode не переходят в целевое состояние.

## Testing Decisions

- Хороший тест проверяет наблюдаемое поведение через публичный интерфейс модуля и не зависит от размещения внутренних файлов, имен приватных hooks или внутреннего состояния.
- Основной seam каждой фичи — ее публичная route-level страница, поднятая с общим session/query/router test harness. Тест взаимодействует с доступным пользователю UI и наблюдает DOM, навигацию и запросы через API test adapter.
- Основной seam `app` — корневое приложение с providers и router. Через него тестируются только redirects, guards, выбор layout и глобальная навигация.
- Чистая детерминированная логика тестируется напрямую только там, где она образует самостоятельный внутренний интерфейс: поиск и выбор действий в `library`, нормализация и validation форм в `admin`.
- Внутренние React-компоненты, hooks и вызовы между ними отдельно не мокируются и не тестируются как самостоятельные публичные контракты.
- Существующий fetch-based API test adapter сохраняется. Переход на MSW не нужен для реализации архитектуры.
- Общий test harness принадлежит `shared` и не импортирует `app` или `features`; конкретная фича сама собирает нужный тестовый router/interface.
- Текущий монолитный integration test является prior art для пользовательских сценариев. Его проверки переносятся к соответствующим фичам и удаляются из старого набора, а не дублируются вторым слоем тестов.
- Существующие Playwright E2E-тесты являются prior art для проверки полного приложения. Они сохраняются и подтверждают неизменность критических auth/navigation сценариев.
- ESLint smoke-checks для ED small должны подтверждать запреты `shared → features/app`, `features → app` и обхода public API, включая разрешение route-level `*.page.tsx`.
- После каждого шага проходят lint, typecheck и unit-тесты. После завершения всей миграции дополнительно проходят production build и E2E.

## Out of Scope

- Синхронизация интерфейса с Pencil.
- Рефакторинг дизайн-системы, CSS tokens и визуальных компонентов.
- Изменение внешнего вида экранов.
- Новая продуктовая функциональность.
- Реализация будущих потоков Заучивания или Повторения.
- Изменение backend или публичного HTTP/API-контракта.
- Ручное изменение generated API artifacts.
- Изменение схемы базы данных или миграции данных.
- Переход на ED medium.
- Добавление верхнеуровневого слоя `services`.
- Замена fetch-based API test adapter на MSW или другой инструмент.
- Изменение package manager, frontend framework, router или query library.

## Further Notes

- Архитектурное решение и причины выбора ED small зафиксированы в ADR-0003.
- Источником текущих правил является `docs/architecture/frontend.md`; ADR и официальная документация Evolution Design объясняют решение, но внешний эталонный репозиторий не является частью контракта.
- `sadhana` служит локальным reference implementation для формы модулей и lint guardrails, но его устаревшие версии зависимостей, MSW/OpenAPI stack и PWA-решения не являются частью контракта.
- Текущий baseline до рефакторинга: frontend typecheck и 24 unit-теста проходят.
- Архитектурная серия выполняется до следующей продуктовой frontend-задачи. Существующая очередь продуктовых задач заблокирована транзитивно через финальную задачу архитектурной серии.
- Реализационная декомпозиция и зависимости находятся в отдельных ticket-файлах. Spec намеренно фиксирует устойчивые решения и не перечисляет конкретные пути файлов.
