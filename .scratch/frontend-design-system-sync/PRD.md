# Синхронизация frontend design system с Pencil

Status: ready-for-human
Accepted: 2026-07-12

## Problem Statement

В проекте сложились две независимые дизайн-системы. В Pencil уже разработан MVP-дизайн с токенами, reusable-компонентами и экранами, а frontend продолжает жить на другой теме, других шрифтах, частично захардкоженных PWA-цветах и локально собранных UI-паттернах внутри фич. Из-за этого новые frontend-задачи могут реализовывать функциональность без привязки к актуальным макетам, а агенты и разработчики вынуждены угадывать визуальные правила по существующему коду.

Владелец продукта хочет, чтобы дизайн был источником истины для пользовательского UI. Новая функциональность с видимым интерфейсом должна ссылаться на макеты и reusable-компоненты, а при расхождении между функциональными требованиями и Pencil-дизайном агент должен остановиться и вернуть коллизию владельцу продукта, а не импровизировать в коде.

## Solution

Сделать `pencil-design.pen` каноническим UI-контрактом для пользовательских экранов, состояний, визуальных паттернов, reusable-компонентов и визуальных токенов. Функциональное поведение остается за PRD, доменными документами, ADR и API-контрактом, но frontend UI считается готовым к реализации только при наличии Pencil references или явно принятого решения по `UI Contract Collision`.

В frontend нужно создать кодовую дизайн-систему, синхронизированную с Pencil: нормализованный контракт токенов, производные CSS/PWA-артефакты, проектные компоненты поверх базовых UI-примитивов и постепенные guardrails от дрейфа. Синхронизация выполняется слоями: сначала workflow и UI-контракт, затем токены и производные артефакты, затем проектные компоненты, затем перенос существующих экранов и пользовательских потоков.

`design/new-design.pen` deprecated.

## Implementation Decisions

- Канонический Pencil-дизайн является UI-контрактом для пользовательских экранов, состояний, визуальных паттернов, reusable-компонентов и визуальных токенов.
- `design/new-design.pen` deprecated.
- Функциональное поведение остается за PRD, доменными документами, ADR и API-контрактом; Pencil определяет визуальную форму, экранные состояния и компоненты.
- UI-задача считается готовой к реализации только если для каждого пользовательского экрана или состояния указаны Pencil references или есть явно принятое решение по `UI Contract Collision`.
- Pencil reference включает человекочитаемое имя узла и nodeId.
- Если nodeId устарел, агент ищет актуальный узел по имени через Pencil MCP и явно отмечает замену ссылки.
- Навигационная карта Pencil-экранов и reusable-компонентов служит индексом для задач и ревью; если карта расходится с Pencil, побеждает Pencil, а карту нужно обновить.
- UI-контрактные коллизии возвращаются владельцу продукта в структурированном формате с требованием, Pencil reference, расхождением, вариантами решения и рекомендацией агента.
- Агент не продолжает реализацию после UI-контрактной коллизии, если выбор меняет поведение, состав экранов, состояния или визуальную систему.
- Мелкие responsive/overflow решения допустимо принимать по существующему Pencil-паттерну, если они не меняют функциональность и состав экрана.
- Issues, которые создают или меняют видимый frontend UI, должны включать Pencil references или явно фиксировать `UI Contract Collision`.
- Финальный отчет по UI-задаче должен включать блок Pencil references со screen, components и exceptions.
- Кодовый контракт токенов живет внутри web-приложения, а не в отдельном workspace package, пока у дизайн-системы один потребитель.
- Структура токенов должна оставаться переносимой, чтобы ее можно было вынести в package при появлении второго потребителя.
- Кодовый контракт токенов нормализует Pencil variables и не копирует все alias/component variables один-в-один.
- Минимальная модель токенов включает reference palette, semantic colors, typography, spacing, radius, elevation и component tokens.
- Каждый code-token имеет явную связь с Pencil variable или source value; токен без такой связи не считается синхронизированным.
- CSS variables, frontend theme, PWA colors, manifest colors и icon colors должны быть производными от code-token contract или проверяться против него.
- Базовые UI-примитивы остаются поверх shadcn/Radix.
- Pencil-синхронизированные проектные компоненты выделяются отдельно от generic UI primitives.
- Pencil `reusable`-узел является UI-контрактом и кандидатом на code component, но не автоматическим требованием создать reusable React-компонент.
- Проектный компонент в `shared/design-system/components` создается только для устойчивого паттерна с повторным использованием, app-level/layout ответственностью, интерактивной/accessibility логикой или component tokens/guardrails, которые не должны копироваться в фичах.
- Если Pencil reusable-компонент выделен только для удобства прототипирования или нужен одному экрану, его можно реализовать локально внутри фичи поверх shadcn/Radix primitives и design tokens; Pencil reference остается обязательным UI-контрактом.
- Полноэкранные Pencil layout-узлы являются reference для визуального состава, отступов и shell-паттерна; они не требуют переносить в код fake mobile status bar, фиксированный phone frame или общий shared layout component.
- Начальные кандидаты на проектные компоненты из Pencil: bottom navigation, shloka card, empty state, page header, review pack, want-to-learn block, streak indicator, settings row и admin form layout. Перед реализацией каждого кандидата нужно классифицировать его как shared project component, локальный feature component или app-level layout pattern.
- Страницы фич передают смысловые данные в уже выделенные проектные компоненты и не повторяют визуальные размеры, цвета, radii, spacing и карточную структуру там, где компонентный контракт уже существует.
- ED small остается архитектурной моделью frontend; design-system слой живет внутри shared и не вводит новый верхнеуровневый слой.
- Синхронизация выполняется слоями: workflow и UI-контракт, токены и производные артефакты, проектные компоненты, затем существующие экраны и пользовательские потоки.
- Автоматические guardrails вводятся после появления code-token contract.
- Минимальные guardrails запрещают новые произвольные hex/OKLCH colors вне token/generator слоя, проверяют PWA colors и постепенно ограничивают magic-size паттерны.
- Нельзя сразу запрещать все Tailwind utility classes в страницах; ограничения ужесточаются по мере переноса UI на проектные компоненты.
- Shadcn configuration должен указывать на фактический CSS entrypoint приложения.
- Существующие route paths, route params, query keys, localStorage keys, API calls, generated API artifacts и публичный HTTP/API-контракт не меняются из-за дизайн-системной синхронизации.
- Backend, схема базы данных и данные не меняются; миграции базы данных не требуются.

## Testing Decisions

- Хороший тест проверяет наблюдаемое поведение и публичный контракт, а не приватную структуру компонентов, имена внутренних helpers или конкретные Tailwind class strings.
- Основной UI seam — route-level frontend сценарии через существующий web test harness. Через него проверяются пользовательские состояния экранов, навигация, доступность действий и русские UI-строки.
- Основной full-app seam — Playwright. Он используется для ключевых экранов и потоков на mobile-first размерах, особенно `390x844` и `360x800`.
- Contract seam для дизайн-токенов проверяет, что CSS variables, frontend theme, PWA colors, manifest colors и icon colors синхронизированы с token contract.
- Guardrail seam проверяет отсутствие новых произвольных hex/OKLCH цветов вне token/generator слоя после введения соответствующего правила.
- Для проектных компонентов тестировать нужно внешний пользовательский результат: отрисовку смысловых props, доступные кнопки/ссылки, выбранное состояние, пустые состояния и переходы.
- Не нужно тестировать приватную композицию проектных компонентов отдельно, если поведение уже покрыто route-level сценарием.
- Для токенов допустимы focused contract/unit tests, потому что token contract является самостоятельным публичным интерфейсом дизайн-системы.
- Для `BottomNavigation` приоритетны route-level или app-level проверки активного раздела, доступности четырех пунктов и переходов.
- Для `ShlokaCard`, `EmptyState`, `ReviewPack` и `WantToLearnBlock` приоритетны проверки через дашборд и библиотеку, где эти компоненты имеют пользовательский смысл.
- Для admin form layout приоритетны route-level admin tests, которые уже проверяют создание и редактирование источника шлоки и шлоки.
- Screenshot/visual checks должны использоваться прагматично для ключевых экранов после стабилизации token/component слоя, а не как единственный способ проверить весь UI.
- Existing frontend tests и Playwright e2e являются prior art; новые проверки должны использовать те же seams вместо создания параллельного test harness.
- Process quality проверяется на уровне issue/PRD acceptance: UI-задачи должны иметь Pencil references или `UI Contract Collision`, а финальные отчеты должны перечислять использованные references.

## Out of Scope

- Создание новых продуктовых возможностей и опережающая реализация их UI или project components до соответствующих функциональных issues.
- Изменение backend, API-контракта, generated API artifacts, схемы базы данных или миграций данных.
- Возврат `design/new-design.pen` в статус UI-контракта.
- Полная одномоментная перепись всех frontend-экранов.
- Переход с ED small на ED medium.
- Добавление верхнеуровневого frontend-слоя для дизайн-системы.
- Вынос design tokens в отдельный workspace package до появления второго потребителя.
- Полный запрет Tailwind utility classes во всех страницах на первом слое синхронизации.
- Автоматическая генерация Pencil variables из репозитория как обязательная часть первого этапа.
- Дорисовка новых Pencil-макетов в рамках этого PRD без отдельного явного решения владельца продукта.
- Создание issues в рамках этого PRD-документа; реализационная декомпозиция выполняется отдельным шагом.
- Desktop-only логика и отдельный desktop design system.
- Замена shadcn/Radix как базового UI implementation layer.
- Изменение доменного словаря `CONTEXT.md` инженерными терминами дизайн-системы.
