# Оформить известные UI Contract Collision для недостающих Pencil-состояний

Status: ready-for-human
Accepted: 2026-07-10

## Родитель

`.scratch/frontend-design-system-sync/spec.md`

## Что сделать

Подготовить проверяемые записи по уже известным расхождениям между функциональными требованиями и Pencil UI-контрактом, чтобы последующие UI-срезы не решали недостающие состояния молча. Срез должен зафиксировать, где дизайн отсутствует или недостаточно точен, какие есть варианты решения для владельца продукта и какие будущие frontend-tickets должны остановиться без принятого решения.

## Критерии приемки

- [x] Для раскрытого состояния пачки повторения после `Показать еще N` оформлен `UI Contract Collision`.
- [x] Для раскрытого состояния списка шлок к заучиванию после `Показать все` оформлен `UI Contract Collision`.
- [x] Для специфичных пустых состояний вкладок `Повторяю` и `Буду учить` оформлен `UI Contract Collision` или явно указано, что используется общий `Product / Empty State`.
- [x] Каждая collision-запись содержит требование, Pencil reference, расхождение, варианты решения и рекомендацию агента.
- [x] Карта Pencil-узлов или связанный task context показывает эти collision как известные блокеры для будущих UI-задач.
- [x] Ни одно недостающее состояние не реализуется в коде в рамках этого среза.
- [x] Последующие ticket-файлы могут ссылаться на эти collision-записи как на проверяемый источник ограничения.

## Результат

Проверяемые записи добавлены в `docs/design/pencil-design-map.md`:

- `UIC-DS-001`: раскрытая пачка повторения после `Показать еще N`.
- `UIC-DS-002`: раскрытый список заучивания после `Показать все`.
- `UIC-DS-003`: пустые состояния вкладок `Повторяю` и `Буду учить`.

Эти записи являются известными блокерами для будущих UI-задач. Последующие
Ticket-файлы могут ссылаться на стабильные collision IDs выше как на источник
ограничения.
Frontend-реализация, пользовательское поведение и Pencil-файл в этом срезе
не менялись.
