# Получать источник шлоки целевым запросом

Status: ready-for-agent

## Родитель

`.scratch/backend-production-hardening/PRD.md` — пользовательские истории 12, 13 и 40.

## Что сделать

Заменить единственную согласованную неэффективную single-record операцию: получение источника шлоки по коду должно выполнять параметризованный целевой SQL, а не загружать и агрегировать все источники с последующим поиском в памяти.

Создание и редактирование шлоки, получение административного источника и остальные single-source сценарии должны использовать этот путь. Формат `Источник шлоки`, порядок частей/глав, validation и внешние API-ответы остаются прежними.

Dashboard, streak, review ranking, полные пользовательские статусы и другие широкие выборки в рамках этой задачи не меняются.

## Критерии приемки

- [ ] Получение источника по коду выполняет параметризованный SQL с predicate по коду и возвращает не более одного агрегированного источника.
- [ ] SQL не строится конкатенацией пользовательского ввода и сохраняет существующее упорядочивание частей и глав источника.
- [ ] Single-source сценарии не вызывают list-all метод для поиска одного источника.
- [ ] Неизвестный код по-прежнему возвращает существующий not-found/validation результат без изменения API-контракта.
- [ ] Создание и редактирование шлоки продолжают проверять принадлежность части и главы выбранному источнику с прежним продуктовым поведением.
- [ ] Repository SQL-shape tests проверяют наличие параметризованного predicate и отсутствие list-all пути, не привязываясь к несущественному форматированию SQL.
- [ ] Существующие catalog/admin tests подтверждают неизменность результата для источника со структурой без глав, с главами и с частями.
- [ ] Dashboard и остальные широкие query flows не рефакторятся попутно; новые индексы и schema migration не добавляются.
- [ ] Существующие API tests, typecheck и build проходят.
