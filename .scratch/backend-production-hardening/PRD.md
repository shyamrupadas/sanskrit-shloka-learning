# Production-hardening бэкенда для первого MVP-релиза

Status: ready-for-agent

## Постановка проблемы

Текущий бэкенд реализует нужные продуктовые сценарии, но его работа с PostgreSQL/Neon и production-конфигурация содержат риски, которые могут проявиться после выкладки на Railway.

Runtime-подключение к Neon использует параметры запуска PostgreSQL, несовместимые с pooled endpoint через PgBouncer. Пул соединений принудительно обновляет подключения каждую минуту, а его размер не согласован с topology одной Railway-реплики. Клиентский timeout не останавливает SQL на сервере, однако некоторые чтения и записи после него автоматически повторяются. Для записи это создает неоднозначный результат: первая попытка может завершиться в PostgreSQL уже после того, как приложение решило повторить операцию.

При прошлых сбоях Neon в проект были добавлены retry и stale-кэши, но точные логи инцидентов не сохранились. Сейчас классификация временных ошибок дублируется в нескольких модулях, а проверка сессии при сбое БД может до пяти минут использовать устаревшие данные. Это повышает доступность ценой некорректной проверки выхода из системы и изменения прав.

Production-запуск также не валидирует конфигурацию целиком, может прочитать локальный dotenv-файл, разрешает любой CORS origin, не включает корректное завершение Nest-приложения и не предоставляет Railway надежный readiness endpoint. Команда миграций зависит от dev-инструмента, эвристически преобразует pooled URL в direct URL и не защищена от параллельного запуска.

Нужна небольшая, сфокусированная итерация по повышению надежности и безопасности. Она не должна менять продуктовые сценарии, вводить ORM, перестраивать все repository-запросы или создавать сложную инфраструктуру, несоразмерную MVP с одним разработчиком и небольшим числом пользователей.

## Решение

Укрепить существующий Nest/`pg` бэкенд на его текущих границах. Runtime будет подключаться к Neon через явно настроенный небольшой пул и pooled URL, а скомпилированный migration runner будет запускаться Railway перед deploy через обязательный direct URL.

Автоматический retry останется только у безопасных чтений и только для классифицированных кратковременных обрывов соединения. Записи и клиентские query timeouts автоматически повторяться не будут. Ошибки доступности БД будут классифицироваться в одном месте и превращаться в контролируемый `503`. Проверка сессии перестанет использовать stale-данные при сбое БД.

Production bootstrap получит строгую env-конфигурацию, точный CORS allowlist, базовые security headers, локальный rate limit для auth endpoints, graceful shutdown, liveness/readiness endpoints и минимальные структурированные логи. Railway build, pre-deploy, start и healthcheck будут зафиксированы как config-as-code.

Из прикладных запросов в этой итерации исправляется только доказанная локальная неэффективность: получение одного источника шлоки не должно загружать все источники. Остальные широкие выборки, schema hardening и инфраструктура real-Postgres тестов остаются будущими улучшениями.

## Решения по реализации

- Работа ограничена backend runtime, database access, migration runner, production bootstrap, Railway-конфигурацией и эксплуатационной документацией. Видимые пользовательские сценарии и HTTP-контракты продукта не меняются.
- Production-топология: статический frontend на VPS по `app.<domain>`, одна долгоживущая Railway-реплика API по `api.<domain>` и текущая БД Neon. Serverless и autoscaling не используются.
- Runtime использует pooled `DATABASE_URL`. Production migration runner требует явный direct `DATABASE_DIRECT_URL`. Эвристическое удаление `-pooler` из hostname удаляется.
- Конфигурация загружается и валидируется в одной небольшой границе без внедрения крупного configuration framework. Проверяются режим окружения, порт, frontend origin и оба URL БД.
- В production dotenv-файлы не читаются. В development и test сохраняется явная поддержка локального dotenv.
- Production origin представляет собой один точный origin без wildcard. Development origin настраивается отдельно и не попадает в production defaults.
- API слушает `0.0.0.0` на порту окружения. Некорректная production-конфигурация завершает процесс до открытия HTTP-порта.
- Runtime pool остается application-side ограничителем поверх Neon PgBouncer. Начальная конфигурация рассчитана на одну небольшую реплику: без постоянного minimum, с явным небольшим maximum около пяти соединений, ограниченным ожиданием подключения и TCP keepalive.
- Принудительный `maxLifetimeSeconds` в 60 секунд удаляется. Размер pool остается настраиваемым, но увеличение требует измеренных queue/latency данных.
- `statement_timeout` и `lock_timeout` не передаются как startup parameters pooled endpoint. Клиентский query timeout не считается серверной отменой SQL и не дает права автоматически повторять запрос.
- TLS-конфигурация следует параметрам Neon connection string. Production-код не копирует из референса `rejectUnauthorized: false`.
- DB-модуль владеет pool lifecycle, логированием pool errors, транзакцией на одном client, классификацией временных connection errors и единой ошибкой недоступности БД.
- У безопасного read query допускается не более одной повторной попытки после классифицированного обрыва соединения. Повтор сопровождается небольшим ограниченным ожиданием и структурированным warning log.
- Query timeout не является retriable error. Ошибки SQL, constraints, неверного запроса и неопределенные сообщения также не повторяются.
- Универсальный `idempotentWriteQuery` удаляется. Записи выполняются один раз; retry возможен только в будущем у отдельной операции с доказанным end-to-end idempotency key и стабильным результатом.
- Repository не определяют DB availability по текстам сообщений. Ожидаемые доменные конфликты продолжают преобразовываться рядом с repository, а инфраструктурная недоступность централизованно преобразуется в `503`.
- Проверка сессии больше не использует stale cache после failed refresh. Короткий fresh cache можно сохранить для MVP, но он должен иметь конечный размер и очищать истекшие записи.
- Ограниченный stale cache разрешен только для некритичных публичных чтений каталога. Auth, authorization и административные чтения не переходят на stale-данные при сбое БД.
- Система сессий не меняется в этой итерации: opaque token продолжает передаваться как Bearer и храниться frontend в `localStorage`.
- CORS разрешает только настроенный production frontend origin, нужные методы и заголовки `Content-Type`/`Authorization`. `credentials` остается выключенным, пока аутентификация использует Bearer header.
- Подключаются базовые Helmet headers. Настройка не вводит frontend CSP-контракт и не меняет ответы API.
- Rate limiting применяется только к публичным login/register endpoints. Для одной реплики счетчики хранятся в памяти; лимиты должны быть достаточно строгими против перебора и достаточно мягкими для обычного входа.
- Express trust proxy настраивается только под ожидаемый Railway proxy path. Безусловное `trust proxy: true` из референса не копируется.
- Добавляются два служебных endpoint: liveness без обращения к БД и readiness с коротким `SELECT 1`. Они не являются новой продуктовой функциональностью и не включаются в пользовательский API-контракт.
- Nest shutdown hooks включаются. При `SIGTERM` приложение перестает принимать новую работу и закрывает pool через существующий lifecycle hook в пределах Railway draining interval.
- Структурированные JSON-логи отправляются в stdout для Railway. HTTP-лог содержит request ID, метод/маршрут, status и duration. DB-лог содержит безопасное имя/категорию операции, duration, error code/category и retry attempt.
- SQL values, authorization headers, session tokens, passwords и персональные данные в логи не записываются. Полный SQL также не является обязательной частью production-логов.
- Pool counters логируются при connection errors, исчерпании/ожидании pool и failed readiness, а не на каждом успешном запросе.
- Получение одного источника шлоки реализуется целевым SQL по коду. Создание шлоки и другие single-source сценарии используют этот метод вместо загрузки всех источников.
- Dashboard, streak, полные списки пользовательских статусов и review summaries в этой итерации не перепроектируются. Их оптимизация требует измеренной проблемы или роста данных.
- Новые ORM/query builder не вводятся. Существующие raw SQL, repository interfaces и transaction boundary сохраняются.
- Новая schema migration для согласованного scope не требуется. Массовое добавление foreign key/check constraints и индексов отложено; существующие примененные миграции не редактируются.
- Migration runner сохраняет упорядочивание, checksums, таблицу истории и одну транзакцию на миграцию.
- Перед чтением истории migration runner получает глобальную advisory-блокировку с ограниченным ожиданием. Потеря соединения автоматически освобождает блокировку.
- Production migration command запускает скомпилированный runner. Ошибка возвращает ненулевой exit code и блокирует Railway deploy.
- Для редкой разрушающей миграции допустимо ручное короткое окно обслуживания. Универсальный expand-contract framework в этой итерации не создается.
- Railway config-as-code фиксирует workspace-aware build, compiled pre-deploy migration command, start command, readiness path и graceful draining settings.
- Первые production-релизы продвигаются вручную после локальных `test`, `typecheck` и `build`. Отдельный CI pipeline не создается.
- Production runbook описывает Railway variables, отличие pooled/direct Neon URL, команды deploy, health endpoints, ручной smoke-checklist, безопасное чтение логов и действия при failed migration.
- Новая ADR не требуется: изменения обратимы, развивают уже выбранные Nest/PostgreSQL/Neon и отдельный backend deployable, не меняя доменную архитектуру.
- Доменный глоссарий не расширяется инфраструктурными терминами. В документации используются существующие термины `источник шлоки`, `шлока`, `статус шлоки пользователя` и `запись повторения`.

## Решения по тестированию

- Хороший тест проверяет наблюдаемое поведение публичной границы модуля: результат, число допустимых попыток, код ошибки, transaction lifecycle или SQL shape. Он не привязывается к private-методам, тексту внутренних логов или порядку несущественных вызовов.
- Основной автоматизированный seam — публичный интерфейс DB-сервиса с fake pool/client. Это существующий seam, уже используемый тестами retry и transaction behavior; новый параллельный database abstraction не создается.
- DB-сервис покрывается сценариями: обычное чтение без retry, один retry после классифицированного connection error, отсутствие retry после timeout/SQL error, отсутствие retry записи, преобразование исчерпанной попытки в DB-unavailable error, освобождение transaction client и закрытие pool.
- Конфигурация pool проверяется как чистое преобразование валидированного env в driver config. Тесты подтверждают небольшой maximum, отсутствие минутного lifetime и отсутствие неподдерживаемых server startup parameters.
- Env loader проверяется как чистая граница над переданным окружением: production требует все значения и не читает dotenv; development допускает локальные defaults; неверный port, URL или origin отклоняется до старта.
- Migration runner тестируется через существующий fake executor. Покрываются первая установка, повторный запуск, checksum mismatch, rollback, advisory lock conflict и ненулевой результат production command при ошибке.
- AuthService тестируется через существующий fake account repository. Покрываются fresh cache, отсутствие stale fallback после transient error, очистка/ограничение cache и сохранение fail-closed поведения authorization.
- HTTP-преобразование DB-unavailable error проверяется на наиболее высокой доступной backend-границе без запуска настоящей БД: клиент получает `503`, а неизвестная ошибка остается безопасным `500`.
- Catalog repository тестируется через существующий fake DatabaseService и SQL-shape seam. Тест подтверждает параметризованный single-source predicate и отсутствие вызова list-all метода для получения одного источника.
- Health behavior тестируется через fake DB dependency: liveness не обращается к БД; readiness возвращает успех после `SELECT 1` и `503` при timeout/connection error без раскрытия деталей.
- Rate limiting, CORS, Helmet, binding к production port и graceful shutdown проверяются сфокусированными bootstrap/HTTP-тестами только там, где это возможно без создания нового тяжелого app harness. Остаток входит в ручной smoke-checklist.
- Manual Railway/Neon smoke-check выполняется после первого deploy: успешный pre-deploy, readiness `200`, разрешенный и запрещенный Origin, login, достижение auth rate limit, одно чтение каталога, одна существующая пользовательская запись и наличие request ID в логах.
- Ручная проверка не должна выводить connection strings, токены или другие секреты в терминал/отчет.
- Существующие API unit tests остаются обязательным regression-набором. Frontend и e2e тесты не меняются, потому что видимого UI и продуктового контракта нет.
- Docker, Testcontainers, локальный PostgreSQL service и автоматические тесты против production Neon не используются.
- Осознанное ограничение: SQL и миграции не проходят отдельный real-Postgres integration test до Railway pre-deploy. Риск ограничивается транзакционными миграциями, checksum history, отменой failed deploy и ручным smoke-check.

## Вне рамок

- Новая продуктовая функциональность или изменение пользовательских сценариев.
- Изменение frontend UI, API product contract или Pencil-макетов.
- Переход с opaque Bearer session на cookie или access/refresh JWT.
- CSRF-защита, необходимая будущему cookie-based session transport.
- Разделение PostgreSQL runtime и migration roles.
- Runtime-валидация body, path и query по TypeSpec или DTO-схемам.
- ORM, query builder или замена `pg`.
- Полная переработка repository interfaces и всех SQL-запросов.
- Оптимизация dashboard, streak, review ranking и полных пользовательских выборок.
- Полный аудит production-данных, массовое добавление foreign key/check constraints или индексов.
- Новая schema migration без найденного критичного дефекта в согласованном scope.
- Универсальные idempotency keys для HTTP-команд.
- Полноценная server-side cancellation framework для произвольных SQL-запросов.
- Docker, Testcontainers, отдельный PostgreSQL service, Neon test branch или real-DB integration suite.
- GitHub Actions или другой новый CI pipeline.
- Автоматический production deploy каждого push.
- Внешний APM, error tracker, metrics backend, alerting или uptime monitor.
- Redis или другое общее хранилище rate-limit state.
- Несколько Railway-реплик, autoscaling или serverless deployment.
- Полный zero-downtime expand-migrate-contract процесс.
- Настройка самого VPS, DNS и TLS frontend за пределами документирования production origin.

## Дополнительные заметки

- Из production-референса принимаются идеи централизованной env-схемы, явного CORS allowlist, binding к `0.0.0.0` и структурированного server logger.
- Из референса не копируются migration runner без истории/checksums/lock, `rejectUnauthorized: false`, hardcoded CORS вместо env, access/refresh JWT без server-side revocation и широкие read-modify-write запросы.
- Neon pooled endpoint работает через PgBouncer в transaction mode. Runtime должен использовать pooled URL, а migration runner — direct URL: https://neon.com/docs/connect/connection-pooling
- Neon PgBouncer принимает только ограниченный набор startup parameters; неподдерживаемые параметры необходимо убрать: https://neon.com/docs/connect/connection-errors
- Railway pre-deploy выполняется после build, получает environment и останавливает deploy при ненулевом exit code: https://docs.railway.com/deployments/pre-deploy-command
- Railway healthcheck допускает новую версию в трафик после `200`, но не является постоянным production monitor после активации: https://docs.railway.com/deployments/healthchecks
- Railway завершает старую версию через `SIGTERM` и configurable draining interval: https://docs.railway.com/deployments/deployment-teardown
- После первого production deploy обязательна отдельная задача по безопасному session transport. Предварительно рекомендуемое направление — сохранить opaque server session, но перенести token в host-only `Secure`, `HttpOnly`, `SameSite=Strict` cookie и добавить CSRF-защиту.
- Также как будущие задачи документируются разделение DB-ролей и строгая runtime-валидация HTTP-входа.
- Переход к shared rate limiting обязателен до запуска второй Railway-реплики. Пересмотр pool size и широких SQL-запросов выполняется по измеренным queue, latency и data-volume показателям.
- Новая schema migration в текущем scope не планируется. Если при реализации все же потребуется новая migration, ее необходимо явно применить через production pre-deploy перед запуском соответствующего кода.
- PRD готов к реализации, поскольку topology, компромиссы доступности/безопасности, тестовые seams и отложенные риски согласованы; открытых продуктовых вопросов нет.
