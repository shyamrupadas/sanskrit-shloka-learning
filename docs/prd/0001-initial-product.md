# Первичный PRD продукта

Status: ready-for-agent

## Постановка проблемы

Пользователь приходит с внешней потребностью выучить заданный набор шлок на санскрите и затем регулярно повторять их по памяти. Без специального инструмента ему трудно видеть, какие шлоки уже выучены, какие нужно выучить, какие нужно повторить сегодня, как прошли прошлые повторения и сохраняется ли ежедневная привычка.

Вторая аудитория - люди, которые любят санскрит и регулярно учат шлоки самостоятельно. MVP оптимизируется под пользователя с обязательным набором шлок, но не должен ломать самостоятельный сценарий.

## Решение

Сделать mobile-first PWA-ready приложение, в котором пользователь может зарегистрироваться, выбрать шлоки из общей библиотеки, добавить их в личный список заучивания, отметить шлоку как выученную и затем повторять ее по алгоритму повторения. Главный экран - дашборд: он показывает компактный индикатор ненулевой серии дней, начальную часть повторения на сегодня и список шлок к заучиванию.

MVP должен включать админку для ручного наполнения общей библиотеки шлок, базовые настройки, статический раздел "Обучение санскриту" и алгоритм повторения на бэкенде. Приложение должно хранить историю завершенных повторений и личные статусы шлок пользователя.

## Решения по реализации

- Создать fullstack-продукт в почти пустом репозитории. Сейчас в репозитории есть документация, `CONTEXT.md` и ADR, но нет кода приложения.
- Использовать термины из `CONTEXT.md` в реализации и UI-текстах. Особенно важны `Пользователь`, `Шлока`, `Заучивание`, `Повторение`, `Шлока в повторении`, `Шлока к повторению сегодня`, `Пачка повторения`, `Буду учить`, `Повторяю`, `Все` и `Тип транслитерации`.
- Следовать ADR 0001: UI MVP только на русском, но тексты интерфейса с самого начала должны быть вынесены в i18n-ключи. Локаль пользователя, локализуемые тексты шлоки и тип транслитерации являются связанными, но разными аспектами отображения.
- Следовать ADR 0002: использовать PostgreSQL как базу данных MVP. Моделировать шлоки, строки, слова, транслитерации, пословные переводы, личные статусы и историю повторений в явной реляционной схеме, используя JSONB только для ограниченных вложенных фрагментов, где это осознанно упрощает MVP.
- Следовать ADR 0003: организовать фронтенд по Evolution Design, применяя подход прагматично и без преждевременной папочной церемонии.
- Следовать ADR 0007: использовать TanStack Router для фронтенд-роутинга. Остальные технологии фронтенда: React, shadcn/ui и TanStack Query.
- Следовать ADR 0006: использовать Nest для бэкенда. Бэкенд отвечает за аутентификацию, прогресс пользователя, права администратора, хранение библиотеки шлок и алгоритм повторения.
- Роутинг должен поддерживать текущую продуктовую карту: дашборд на `/`, библиотека на `/shlokas`, страница шлоки по коду, поток заучивания по коду, поток повторения по коду, раздел "Обучение санскриту", настройки, auth-экраны и защищенный административный маршрут шлок.
- Фронтенд должен быть mobile-first с нижней навигацией по четырем основным разделам: дашборд, библиотека шлок, "Обучение санскриту" и настройки. В нижней навигации допустима короткая подпись `Обучение`; в заголовках и документации каноническое название раздела остается `Обучение санскриту`.
- Приложение должно быть PWA-ready: манифест, иконки, адаптивная верстка. Офлайн-синхронизация прогресса, push-уведомления и UX установки не входят в MVP.
- Регистрация использует email, пароль и подтверждение пароля на форме. Вход использует email и пароль. Гостевого режима нет. Личный прогресс существует только внутри учетной записи.
- Формы регистрации и входа включают действие `Показать пароль`. На регистрации оно управляет полями пароля и подтверждения пароля вместе; на входе - полем пароля. Включенное действие показывает введенный текст открыто, выключенное скрывает значение.
- Ошибка неверных данных на входе должна быть общей: `Неверный email или пароль`. Она не должна отдельно подтверждать существование email или неверность только пароля.
- Политика сложности пароля для MVP: минимум 6 символов. Это правило должно проверяться и на фронтенде, и на бэкенде. Другие требования к составу пароля в MVP не вводятся.
- Роли простые: администратор - дополнительная роль/право пользователя, а не отдельный тип учетной записи. Администратор также может иметь личный прогресс.
- В настройках MVP есть тип транслитерации, hard mode, logout и вход в админку для администратора. Настройка локали пользователя в MVP не показывается.
- Доменная модель шлоки должна поддерживать уникальный неизменяемый код шлоки, источник шлоки, номер шлоки, отображаемое название шлоки, опциональный полный перевод шлоки и статус публикации.
- Текстовая модель шлоки должна состоять из строк и слов. У каждого слова обязательны кириллическая транслитерация (`ru`), латинская транслитерация (`en`) и один русский пословный перевод, общий для обоих типов транслитерации в MVP.
- В будущей английской версии пословный перевод и другие локализуемые тексты шлоки должны возвращаться с бэкенда на языке локали пользователя, а не выбираться фронтендом из всех доступных языков. В MVP бэкенд может возвращать только русские локализуемые тексты шлоки, сохраняя тот же контракт ответа.
- Деванагари не требуется в MVP.
- `Код шлоки` уникален и неизменяем после создания. Пара `источник шлоки + номер шлоки` также уникальна.
- В библиотеке есть опубликованные и скрытые шлоки. Опубликованные шлоки доступны для нового добавления. Скрытые шлоки исчезают из списков "буду учить", но остаются доступными пользователям, которые уже выучили их.
- Управление шлоками в админке включает создание, редактирование, публикацию и скрытие. Физического удаления и пакетного импорта в MVP нет.
- Редактирование канонического текста шлоки обновляет его для всех пользователей и не сбрасывает прогресс. Версионирование текста не является продуктовой возможностью.
- Библиотека на фронтенде имеет три таба: `Повторяю`, `Буду учить` и `Все`. По умолчанию открывается таб `Повторяю`.
- Пустые состояния обязательны для вкладок `Повторяю` и `Буду учить`.
- Поиск в табе `Все` работает по отображаемому названию шлоки, источнику шлоки и номеру шлоки. В MVP поиск может быть реализован на фронтенде; бэкенд-поиск можно добавить позже.
- Таб `Все` показывает шлоки в стабильном порядке по источнику шлоки и номеру шлоки. Пользовательские контролы сортировки и фильтров не входят в MVP.
- Карточка шлоки показывает отображаемое название шлоки, короткий фрагмент текста шлоки в выбранном типе транслитерации, статус и действие. Источник шлоки и номер шлоки не нужно дублировать отдельно, если они уже входят в отображаемое название шлоки.
- Базовые MVP-экраны одного пользователя показывают шлоки в дефолтном типе транслитерации `ru`; латиница показывается только в варианте UI, где явно демонстрируется переключенная настройка типа транслитерации.
- Карточки библиотеки показывают статусное действие по активной вкладке: во вкладке `Все` для доступной шлоки - `Добавить`; во вкладке `Буду учить` - `Учить`; во вкладке `Повторяю` - `Повторить`. Переход по телу карточки или chevron открывает минимальную страницу шлоки.
- `Статус шлоки пользователя` имеет ровно три состояния в MVP: `доступна`, `буду учить` и `повторяю`.
- Добавление шлоки из таба `Все` создает личное состояние `буду учить`. Удаление из "буду учить" возвращает шлоку в `доступна`.
- Отметка шлоки как выученной сразу переводит ее в `повторяю`. Подтверждение преподавателя или тест не нужны.
- Если шлока стала `повторяю`, пользователь не может убрать ее из повторения.
- Неудачные попытки заучивания не сохраняются как подробная история в MVP. Нужно сохранить время перевода шлоки в `повторяю`; подробная история заучивания остается будущей доработкой.
- Поток повторения сохраняет только завершенные записи. Выход до результата не влияет на историю, сегодняшних кандидатов и серию дней.
- Завершенная запись повторения должна включать пользователя, шлоку, время завершения, день пользователя и один из четырех результатов: `вспомнил без ошибки`, `вспомнил с ошибкой`, `вспомнил с подсказкой`, `не смог вспомнить`.
- Завершенное повторение учитывается независимо от точки входа MVP: дашборд или библиотека.
- Последовательность подсказок в MVP фиксирована: первая половина первой строки, вся первая строка, затем весь текст. Открытие полного текста после неудачного вспоминания записывает `не смог вспомнить`.
- Если пользователь сначала вспомнил шлоку сам, после показа полного текста он обязан оценить результат как правильный или с ошибкой, прежде чем повторение считается завершенным.
- Бэкенд владеет алгоритмом повторения. Он выбирает кандидатов из шлок в статусе `повторяю`, исключая шлоки, повторенные в текущий день пользователя, и шлоки, переведенные в повторение в текущий день пользователя.
- Алгоритм сортирует кандидатов по приоритетам: последний результат `не смог вспомнить`; последний результат `вспомнил с ошибкой`; последний результат `вспомнил с подсказкой`; шлока переведена в статус `повторяю` менее 5 дней назад; шлока не повторялась более 5 дней; все остальные шлоки в статусе `повторяю`.
- Внутри одного приоритета раньше показываются шлоки, которые дольше не повторялись. Для шлок без истории повторений используется дата перевода в повторение.
- Бэкенд для начального дашборда возвращает первые 5 кандидатов как начальную пачку повторения и количество оставшихся кандидатов на день.
- При действии `Показать еще N` фронтенд отправляет запрос на бэкенд, а бэкенд возвращает все оставшиеся неповторенные сегодня шлоки, исключая уже показанные и уже повторенные сегодня.
- Ручное повторение любой шлоки в статусе `повторяю` уменьшает оставшееся количество, если эта шлока еще не была повторена в текущий день пользователя.
- UI не должен объяснять пользователю, почему шлока попала в пачку повторения. Причины алгоритма скрыты от пользователя продуктовым решением.
- Дашборд показывает начальную пачку повторения, компактную ссылку `Показать еще N` при наличии остатка, до 3 шлок к заучиванию, пустые состояния и компактный индикатор ненулевой серии дней.
- Действие `Показать все` в блоке шлок к заучиванию раскрывает весь список шлок для заучивания прямо на дашборде.
- На пустом дашборде нового пользователя скрываются компактный индикатор серии дней и блок повторения; первый экран ведет к одному действию `Добавить`, которое открывает библиотеку для выбора первых шлок.
- Отдельный заголовок `Сегодня` на дашборде не обязателен.
- После завершения всех кандидатов на день нужно показать встроенное состояние на дашборде, а не модальное окно.
- `Серия дней` считается по дню пользователя. Значимая активность - завершенное повторение или перевод шлоки в `повторяю`. Серия увеличивается не более одного раза в день, сбрасывается после пропущенного дня и отображается на дашборде только если она ненулевая.
- Индикатор серии дней показывает состояние серии дней: серый вид означает, что ненулевая серия еще ожидает значимой активности в текущий день пользователя; активный вид означает, что серия уже продолжена сегодня.
- Страница шлоки в MVP намеренно минимальная: отображаемое название шлоки, текст шлоки, возврат в библиотеку и нижняя навигация общего мобильного shell. Статус пользователя, пословная транслитерация, пословный перевод, полный перевод и действия по статусу остаются вне текущей версии страницы шлоки.
- Статический раздел "Обучение санскриту" - простой список раскрывающихся статей-советов. CMS для этого раздела в MVP нет. Конкретные тексты советов не являются продуктовым блокером: в первой реализации допустим временный авторский текст разработчика, который будет отредактирован позже.
- Основные фронтенд-модули: auth, dashboard, shloka library, shloka detail, learn shloka, review shloka, Sanskrit learning, settings, admin shlokas, общая оболочка приложения и навигация, общий i18n и общий API-клиент.
- Основные бэкенд-модули: authentication/accounts, authorization/roles, shloka catalog, user shloka progress, repetition history, repetition algorithm, streak calculation, settings/preferences и admin shloka management.
- Глубокие модули, которые стоит изолировать за стабильными интерфейсами: алгоритм повторения, расчет серии дней, переходы статуса шлоки пользователя, валидация шлоки и выбор отображаемой транслитерации. Эти модули не должны зависеть от деталей UI-фреймворка.

## Решения по тестированию

- Тесты должны проверять внешнее поведение и доменные результаты, а не внутренние детали компонентов или базы данных.
- Алгоритм повторения требует сфокусированных unit-тестов как глубокий модуль с высоким продуктовым риском. Нужно покрыть исключение шлок, повторенных сегодня, исключение шлок, выученных сегодня, все уровни приоритета, сортировку внутри приоритета, начальный лимит 5, оставшееся количество и выдачу всех оставшихся шлок по `Показать еще N`.
- Расчет серии дней требует unit-тестов. Нужно покрыть первую активность, несколько активностей в один день, последовательные дни, сброс после пропуска, границы локального дня пользователя и состояние серии дней до/после первой значимой активности текущего дня.
- Переходы статуса шлоки пользователя требуют unit-тестов. Нужно покрыть `доступна -> буду учить`, `буду учить -> доступна`, `буду учить -> повторяю` и запрет удаления шлоки из `повторяю`.
- Валидация шлоки требует unit-тестов. Нужно покрыть уникальный код, неизменяемый код после создания, уникальность источника и номера, обязательные метаданные, обязательную структуру строк и слов, обязательные `ru` и `en` для каждого слова, обязательный русский пословный перевод для каждого слова, опциональный полный перевод и отсутствие обязательного деванагари.
- Поведение скрытия/публикации в админке требует тестов. Нужно покрыть исчезновение скрытых шлок из "буду учить", сохранение скрытых выученных шлок в статусе `повторяю` и участие скрытых выученных шлок в алгоритме.
- Поток повторения нужно покрыть интеграционными или end-to-end тестами. Нужно проверить все четыре результата повторения и убедиться, что незавершенные повторения не пишут историю.
- Поток заучивания нужно покрыть интеграционными или end-to-end тестами. Нужно проверить, что "Не выучил" оставляет шлоку в "буду учить", а "Выучил" переводит ее в `повторяю` и создает значимую активность.
- Аутентификацию нужно покрыть интеграционными тестами на регистрацию с подтверждением пароля, минимальную длину пароля 6 символов на фронтенде и бэкенде, общее сообщение ошибки входа, действие `Показать пароль`, вход, выход и поведение защищенных маршрутов.
- Настройки нужно покрыть тестами на сохранение и чтение типа транслитерации и hard mode, а также на то, что hard mode не влияет на MVP-алгоритм.
- Поведение фронтенда нужно тестировать на уровне пользовательских сценариев там, где это практично: пустой дашборд нового пользователя без серии дней и блока повторения, раскрытие `Показать еще N`, раскрытие `Показать все` для шлок к заучиванию, пустые состояния вкладок `Повторяю` и `Буду учить`, минимальная страница шлоки, карточные действия `Добавить`/`Учить`/`Повторить`, отображение базовых экранов в типе транслитерации `ru`, добавление шлоки из библиотеки, заучивание шлоки, повторение шлоки, создание/редактирование/скрытие/публикация шлоки администратором.
- Так как сейчас в репозитории нет кода приложения и существующих тестов, тестовый каркас нужно добавить вместе с первым вертикальным срезом.

## Вне рамок

- Вход через Google.
- Подтверждение email.
- Восстановление пароля.
- Гостевой режим.
- Push-уведомления и напоминания.
- Офлайн-прогресс и разрешение конфликтов синхронизации.
- UX установки приложения и явные сценарии "добавить на главный экран".
- Полноценный английский интерфейс.
- Пользовательский выбор языка интерфейса в MVP.
- Курсы, квизы и структурированные обучающие программы внутри "Обучения санскриту".
- CMS или админское редактирование статических советов в "Обучении санскриту" в MVP.
- Подробная история попыток заучивания.
- Пользовательские шлоки.
- Монетизация, подписки, free/pro лимиты и дневные лимиты использования.
- Мотивационные реакции за 2, 3 или больше выученных шлок в день.
- Пользовательский экран истории повторений.
- Расширенная страница шлоки: статус пользователя, пословная транслитерация, пословный перевод, полный перевод и действия по статусу `Добавить`, `Начать заучивание` или `Повторить`.
- Пакетный импорт шлок в админке.
- Редактирование профиля пользователя, отображаемое имя, аватар или управление email.
- Пользовательское объяснение, почему шлока попала в пачку повторения. Это не планируется как продуктовая возможность.
- Версионирование текста шлоки. Это не планируется как продуктовая возможность.
- Физическое удаление шлок из общей библиотеки.
- Фильтры по источнику шлоки и пользовательское управление сортировкой в общей библиотеке.
- Бэкенд-поиск в MVP, если фронтенд-поиска достаточно для начального размера библиотеки.

## Дополнительные заметки

- Начальные seed-шлоки намеренно не указаны в PRD. Администратор вручную наполнит общую библиотеку, когда MVP-инструменты админки будут готовы.
- Недостающие макеты для пустых состояний вкладок `Повторяю` и `Буду учить`, раскрытого списка повторения после `Показать еще N` и раскрытого списка шлок для заучивания после `Показать все` дорисовываться не будут. Разработка должна реализовать эти состояния по уже зафиксированным продуктовым правилам и существующему визуальному направлению.
- Уведомления - высокоприоритетная доработка после MVP.
- Подробная история заучивания - доработка после MVP.
- Пользовательские шлоки - доработка после MVP.
- Монетизация и free/pro лимиты - доработка после MVP.
- Форма backend contract artifacts для TypeSpec/OpenAPI pipeline зафиксирована в ADR 0005.
- Согласованный контекст зафиксирован в `CONTEXT.md`; архитектурные решения зафиксированы в ADR 0001-0007.
