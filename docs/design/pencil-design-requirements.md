# Бриф для Pencil: дизайн MVP

## Общая задача

Сделать mobile-first дизайн приложения для заучивания и повторения шлок на санскрите. Интерфейс на русском. Первый экран после входа - рабочий дашборд, не лендинг.

Стиль: спокойный учебный инструмент для ежедневной практики. Без игрового давления, тяжелой религиозной стилизации, декоративных hero-блоков и лишних визуальных эффектов. Главное - читаемость, ясная навигация и быстрый ежедневный сценарий на телефоне.

Основной размер макетов: `390 x 844`. Дополнительно проверить `360 x 800`. Desktop-адаптацию подготовить позже, без отдельной desktop-only логики.

## Общие UI-правила

- Нижняя мобильная навигация: `Дашборд`, `Библиотека`, `Обучение`, `Настройки`. Короткая подпись `Обучение` допустима только в навигации; каноническое название раздела - `Обучение санскриту`.
- Использовать термины: `шлока`, `заучивание`, `повторение`, `серия дней`, `буду учить`, `повторяю`.
- Не использовать: `стих`, `тренировка`, `streak`. Разговорное `аккаунт` допустимо только в коротких фразах входа и регистрации.
- Карточка шлоки показывает отображаемое название шлоки, короткий фрагмент текста шлоки кириллицей, статус и действие. Источник/номер не нужно дублировать отдельно, если они уже входят в отображаемое название шлоки.
- Базовые макеты одного пользователя показывают шлоки кириллицей. Латинская транслитерация остается будущей доработкой.
- Тексты должны помещаться на мобильном экране, длинные названия и слова должны переноситься без поломки верстки.
- Пустые состояния и ошибки нужно показать в макете, но без избыточной детализации.

## Этап 1: регистрация, вход, дашборд

Цель этапа: получить первые экраны, по которым можно согласовать базовый стиль, навигацию и главный ежедневный сценарий.

### 1. Регистрация

Нужен экран создания учетной записи.

Элементы:

- заголовок;
- поле email;
- поле пароль;
- поле подтверждения пароля;
- кнопка регистрации;
- ссылка на вход;
- место для ошибки формы.

Подтверждение пароля проверяется на форме регистрации. Политика сложности пароля для MVP: минимум 6 символов. Это правило должно проверяться и на фронтенде, и на бэкенде; другие требования к составу пароля в MVP не вводятся.

Действие `Показать пароль` является частью MVP-формы регистрации. Если оно включено, поля пароля и подтверждения пароля показывают введенный текст открыто; если выключено, оба поля скрывают значение.

Не проектировать вход через Google, гостевой режим, подтверждение email и восстановление пароля.

### 2. Вход

Нужен экран входа в учетную запись.

Элементы:

- поле email;
- поле пароль;
- кнопка входа;
- ссылка на регистрацию;
- состояние ошибки неверных данных.

Ошибка неверных данных на входе должна быть общей: `Неверный email или пароль`. Формулировка не должна отдельно подтверждать существование email или неверность только пароля.

Действие `Показать пароль` является частью MVP-формы входа. Если оно включено, поле пароля показывает введенный текст открыто; если выключено, поле скрывает значение. Макет не должен показывать отмеченное действие `Показать пароль` одновременно с замаскированным паролем.

После успешного входа пользователь попадает на дашборд.

### 3. Дашборд

Главный экран ежедневной работы.

Обязательные элементы:

- компактный индикатор `Серия дней` в верхней зоне дашборда;
- блок повторения с начальными 1-5 шлоками;
- ссылка `Показать еще N`, если после начальных шлок есть другие шлоки к повторению сегодня;
- список шлок для заучивания, сначала до 3 шлок;
- действие `Показать все`, раскрывающее весь список шлок для заучивания прямо на дашборде;
- нижняя навигация из четырех разделов.

Отдельный заголовок `Сегодня` не обязателен: на мобильном дашборде приоритетнее быстро показать серию дней, повторение и заучивание.

Индикатор `Серия дней` показывает состояние серии дней: серый вид означает, что ненулевая серия еще ожидает значимой активности в текущий день пользователя; активный огненный вид означает, что серия уже продолжена сегодня. Это два состояния одного счетчика, а не отдельные счетчики.

Нужно показать состояния:

- пустой дашборд нового пользователя с переходом в библиотеку;
- обычный дашборд со шлоками к повторению и заучиванию;
- все повторения на сегодня завершены;
- раскрытый список повторения после `Показать еще N`, где показаны все оставшиеся шлоки к повторению сегодня.

На пустом дашборде нового пользователя скрываются компактный индикатор серии дней и блок повторения. Первый экран должен вести к одному действию `Добавить`, которое открывает библиотеку для выбора первых шлок.

Важное ограничение: дашборд не должен давить количеством оставшихся повторений. Остаток показывать через компактную ссылку `Показать еще N`; при клике приложение запрашивает и показывает все оставшиеся неповторенные сегодня шлоки, а не только следующую порцию.

В `design/pencil-design.pen` раскрытое состояние после `Показать еще N` пока не отрисовано и дорисовываться перед разработкой не будет. Реализация должна следовать правилам выше и существующему визуальному направлению.

В `design/pencil-design.pen` раскрытое состояние списка шлок для заучивания после `Показать все` пока не отрисовано и дорисовываться перед разработкой не будет. Реализация должна следовать правилам выше и существующему визуальному направлению.

### Примерные данные для этапа 1

- `Бхагавад-гита 1.1`
- `Бхагавад-гита 1.2`
- `Иша-упанишада 1`

Примеры CTA:

- `Повторить`
- `Учить`
- `В библиотеку`
- `Показать все`
- `Показать еще N`

`Учить` допустимо только как короткая CTA-команда, которая запускает заучивание. Для длинного текста использовать `Начать заучивание`; не использовать `Начать обучение` для действий со шлокой.

## Этап 2: остальные экраны

Цель этапа: после согласования этапа 1 подготовить полный набор экранов MVP.

### Библиотека шлок

Табы:

- `Повторяю` - открыт по умолчанию;
- `Буду учить`;
- `Все`.

Нужны карточки шлок, пустые состояния и состояние без результатов поиска. Поиск в MVP показывается только во вкладке `Все` и ищет по общей библиотеке. Пустые состояния обязательны для вкладок `Повторяю` и `Буду учить`: пользователь должен понимать, почему список пуст и куда перейти дальше.

В `design/pencil-design.pen` пустые состояния вкладок `Повторяю` и `Буду учить` пока не отрисованы и дорисовываться перед разработкой не будут. Реализация должна сделать понятные пустые состояния по продуктовым правилам и существующему визуальному направлению.

Карточки в библиотеке показывают статусное действие для текущего таба:

- в `Все` для доступной шлоки - `Добавить`;
- в `Буду учить` - `Учить`;
- в `Повторяю` - `Повторить`.

Переход по телу карточки или chevron открывает минимальную страницу шлоки.

### Страница шлоки

В MVP страница шлоки намеренно минимальная. Показать:

- название;
- текст шлоки;
- возврат в библиотеку;
- нижнюю навигацию общего мобильного shell.

Нижняя навигация на странице шлоки не считается расширенным статусным действием и не нарушает минимальность страницы.

#### Вне рамок текущей страницы шлоки

- статус пользователя;
- пословная транслитерация;
- пословный перевод;
- полный перевод;
- действия по статусу: `Добавить`, `Начать заучивание` или `Повторить`.

Эти элементы будут доработаны отдельной задачей после согласования минимального MVP-экрана.

### Заучивание

Экран должен показывать шлоку, переводы и действия:

- `Выучил`;
- `Не выучил`;
- ссылка на советы в `Обучении`.

После `Выучил` нужен экран подтверждения с действиями:

- `Выучить еще`;
- `На дашборд`.

### Повторение

Нужны состояния потока:

1. Текст скрыт, пользователь пробует вспомнить шлоку.
2. Первая подсказка: половина первой строки.
3. Вторая подсказка: вся первая строка.
4. Полный текст.
5. Выбор результата после самостоятельного вспоминания:
   - `Все правильно`;
   - `Сделал ошибку`.

Также нужны действия:

- `Открыть подсказку`;
- `Показать весь текст`;
- переход к следующей шлоке или завершение потока.

### Обучение санскриту

Простой экран со списком раскрывающихся советов. Без курсов, квизов и прогресса. Конкретные тексты советов не являются дизайн-блокером: в первой реализации допустим временный авторский текст разработчика, который будет отредактирован позже.

Пример секций:

- `Как читать шлоку по строкам`;
- `Как учить через смысл слов`;
- `Что делать, если шлока не вспоминается`.

### Настройки

Показать:

- переключатель `Hard mode`;
- `Выйти из учетной записи`;
- административные действия `Добавить источник` и `Добавить шлоку` только для администратора.

### Админка текущего этапа

На текущем этапе кнопки `Добавить источник` и `Добавить шлоку` размещаются в разделе `Настройки` и видны только администратору.

Нужны базовые формы:

- создание источника;
- редактирование источника;
- создание шлоки;
- редактирование шлоки.

Форма источника:

- код источника;
- название источника;
- описание источника;
- структура источника: `Без глав`, `Главы`, `Части`;
- для структуры `Главы`: список названий глав и действие `Добавить главу`;
- для структуры `Части`: название части, вложенные главы, действия `Добавить главу` и `Добавить часть`.

Форма шлоки:

- выбор источника;
- выбор главы, если у источника есть главы;
- поле `Текст` для номера шлоки внутри выбранного места источника;
- поля `1 пада`, `2 пада`, `3 пада`, `4 пада`;
- перевод шлоки, опционально.

Созданная шлока сразу доступна всем пользователям в общей библиотеке. Не проектировать список админки, публикацию/скрытие, физическое удаление, пакетный импорт, латинскую транслитерацию и пословные переводы на текущем этапе.

## Что отдать после каждого этапа

После этапа 1:

- mobile-макеты регистрации, входа и всех состояний дашборда;
- нижняя навигация;
- базовые компоненты: кнопки, поля, карточка шлоки, бейдж статуса, пустое состояние.

После этапа 2:

- mobile-макеты всех остальных экранов;
- состояния основных потоков;
- desktop-адаптация ключевых экранов;
- краткие правила дизайн-системы: цвета, типографика, кнопки, табы, карточки, статусы.
