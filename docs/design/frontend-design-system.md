# Frontend Design System

`design/pencil-design.pen` является визуальным источником истины для frontend UI. Кодовая дизайн-система в `apps/web` должна быть синхронизированным машинно-читаемым отражением Pencil-дизайна, а не независимой темой приложения.

## Источники истины

- Функциональное поведение: PRD, доменный словарь, ADR и API-контракт.
- Визуальный UI-контракт: `design/pencil-design.pen`.
- Навигационный индекс Pencil-узлов: `docs/design/pencil-design-map.md`.
- Кодовый контракт токенов: `apps/web/src/shared/design-system/tokens/`.

Если кодовый контракт токенов или markdown-индекс расходится с `design/pencil-design.pen`, каноническим источником считается `.pen`, а производные артефакты нужно обновить перед frontend-реализацией.

## Готовность UI-задач

Требование ссылаться на Pencil применяется ко всем задачам, которые создают или меняют видимый пользовательский UI: frontend-реализация, route-level flows, e2e-сценарии и acceptance criteria для пользовательских экранов.

Backend/API-only задача не обязана ссылаться на Pencil, если она не вводит новый видимый пользовательский сценарий. Если backend-задача вводит новый пользовательский сценарий, родительский PRD/issue должен иметь связанный UI-contract или явно фиксировать, что UI будет отдельной задачей и еще не готов к реализации.

## Коллизии UI-контракта

Если функциональное требование и Pencil расходятся, агент не должен сам выбирать вариант и продолжать реализацию, если выбор меняет поведение, состав экранов, состояния или визуальную систему. Агент возвращает владельцу продукта structured report:

```md
## UI Contract Collision

- Требование: ...
- Pencil reference: "..." (...)
- Расхождение: ...
- Варианты решения:
  1. Обновить Pencil: ...
  2. Изменить требование: ...
  3. Разрешить design exception: ...
- Рекомендация агента: ...
```

Явные мелкие responsive/overflow детали можно решать по существующему Pencil-паттерну, если экранный состав и функциональное поведение не меняются.

## Отчет по UI-задаче

Если задача затрагивала видимый UI, финальный отчет агента должен включать краткий блок:

```md
Pencil references:
- Screen: "..." (...)
- Components: "..." (...), "..." (...)
- Exceptions: none
```

Если были расхождения, вместо `Exceptions: none` нужно указать ссылку на принятое владельцем продукта решение.

## Размещение токенов

Пока единственный потребитель дизайн-системы — `apps/web`, канонический repo-контракт токенов живет внутри frontend-приложения:

```text
apps/web/src/shared/design-system/tokens/
```

Отдельный workspace package для дизайн-токенов сейчас не вводится. Структуру токенов нужно держать переносимой, чтобы при появлении второго потребителя ее можно было вынести в `packages/*` без смены модели токенов.

## Модель токенов

Кодовый контракт не должен один-в-один копировать все Pencil variables. В Pencil допустимы alias- и component-переменные, удобные для макета; в коде нужна нормализованная модель без дублей вроде `primary`, `color-brand`, `button-primary-bg`, `bg` и `color-bg-app` как независимых смыслов.

Минимальная структура токенов:

- reference palette;
- semantic colors;
- typography;
- spacing;
- radius;
- elevation;
- component tokens для устойчивых проектных компонентов.

Каждый code-token должен иметь явную связь с Pencil variable или source value. Если связь неясна, токен нельзя считать синхронизированным с Pencil.

## Компонентная граница

Базовые примитивы поверх shadcn/Radix остаются в:

```text
apps/web/src/shared/ui/
```

Pencil-синхронизированные проектные компоненты живут отдельно:

```text
apps/web/src/shared/design-system/components/
```

Эта граница нужна, чтобы не смешивать generic UI primitives и продуктовую дизайн-систему Sanskrit Shloka Learning. Страницы фич должны собираться из проектных компонентов и передавать смысловые props, а не повторять Tailwind-размеры, цвета, radii, spacing и карточную структуру.

Начальный набор проектных компонентов должен соответствовать reusable-компонентам и устойчивым экранным паттернам из `design/pencil-design.pen`: `BottomNavigation`, `ShlokaCard`, `EmptyState`, `PageHeader`, `ReviewPack`, `WantToLearnBlock`, `StreakIndicator`, `SettingsRow`, `AdminFormLayout`.

## Подход к синхронизации

Синхронизация текущего frontend с Pencil выполняется слоями: сначала агентский workflow и UI-контракт, затем токены и производные артефакты, затем проектные компоненты, затем экраны и пользовательские потоки. Детальная декомпозиция на PRD и issues в этом документе не ведется.

## Guardrails от дрейфа

Автоматические проверки нужно вводить после появления code-token contract, а не раньше. Минимум проверок:

- запрет новых произвольных hex/OKLCH цветов в `apps/web/src` вне token/CSS generator слоя;
- проверка PWA `theme_color`, manifest colors и icon colors против design tokens;
- проверка прямых magic-size паттернов сначала для новых проектных компонентов.

Не нужно сразу запрещать все Tailwind utility classes в страницах. Ограничения ужесточаются постепенно по мере переноса UI на `shared/design-system/components`.
