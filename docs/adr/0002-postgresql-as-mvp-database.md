# Использовать PostgreSQL как хранилище MVP

Для MVP используем PostgreSQL как основное хранилище данных. Production-совместимым managed вариантом является Neon, локальная разработка должна подключаться через `DATABASE_URL` из `.env`/`.env.local`. Структуру шлоки, личный прогресс, историю повторений и учетные записи моделируем реляционно, а вложенные фрагменты текста шлоки раскладываем на связанные таблицы или на явно ограниченные JSONB-поля там, где это упрощает MVP без размывания доменной модели.

**Рассмотренные варианты**

- Использовать PostgreSQL и нормализовать учетные записи, роли, шлоки, слова, переводы, личные статусы и историю повторений.
- Использовать MongoDB и хранить вложенные структуры шлок и связанные пользовательские данные в документной модели.

**Последствия**

PostgreSQL делает связи между пользователем, личной шлокой, статусом заучивания, повторениями и административными правами явными с самого начала. Это полезно для будущих отчетов, уникальных ограничений, миграций и managed Postgres hosting через Neon. Цена решения - нужно раньше определить схему таблиц и миграции; вложенную текстовую модель шлоки нельзя оставлять неструктурированной без осознанного решения по границам JSONB.
