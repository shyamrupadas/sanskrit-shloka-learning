# Разделить локаль пользователя и тип транслитерации

MVP выходит с русским интерфейсом, но тексты интерфейса нужно с самого начала выносить в i18n-ключи, потому что позже ожидается поддержка английской локали. Локаль пользователя, локализуемые тексты шлоки и тип транслитерации текста шлоки являются разными аспектами отображения: русская локаль по умолчанию использует кириллическую транслитерацию, английская локаль по умолчанию использует латинскую транслитерацию, а пользователь может переопределить тип транслитерации независимо. Пословные переводы и другие локализуемые тексты шлоки возвращает бэкенд на языке локали пользователя.

**Рассмотренные варианты**

- Захардкодить русские тексты интерфейса в MVP и перейти на i18n позже.
- Сразу вынести тексты интерфейса в i18n, даже если MVP поставляется только с русским интерфейсом.
- Считать пословные переводы частью фронтенд-i18n.
- Считать пословные переводы локализуемым содержимым шлоки, которое выбирает и возвращает бэкенд.
- Возвращать с бэкенда все языковые варианты локализуемых текстов шлоки и выбирать нужный вариант на фронтенде.
- Возвращать с бэкенда только локализуемые тексты шлоки, уже выбранные по локали пользователя.

**Последствия**

Это добавляет небольшую начальную стоимость реализации, но убирает будущую миграцию по всему фронтенду и сохраняет продуктовое правило: локаль пользователя не полностью определяет, как отображается текст шлоки. В MVP бэкенд может возвращать только русские локализуемые тексты шлоки, сохраняя тот же контракт ответа. Когда появится английская версия, бэкенд должен будет учитывать локаль пользователя при выдаче локализуемых текстов шлоки, а не перекладывать выбор языкового варианта на фронтенд.
