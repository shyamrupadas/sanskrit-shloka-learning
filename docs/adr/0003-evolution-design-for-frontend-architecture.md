# Использовать Evolution Design small для архитектуры фронтенда

Для `apps/web` используем [Evolution Design small](https://ed.evocomm.space/guide/ed-small/). Верхнеуровневые слои фронтенда — `app`, `features` и `shared`: `app` запускает приложение и связывает фичи, `features` содержит крупные пользовательские возможности, а `shared` — действительно глобальные UI-компоненты, API-инфраструктуру, модели и утилиты. Отдельных верхнеуровневых слоев `pages` и `services` в текущей модификации нет; `services` появится только при переходе к ED medium, если возникнет доказанная потребность в переиспользуемых бизнес-модулях между фичами.

Направление зависимостей контролируется ESLint: `shared` не импортирует `features` или `app`, а `features` не импортирует `app`. В ED small прямые импорты между фичами допустимы, но должны быть редкими и проходить через public API модуля. Внутренняя структура каждого модуля эволюционирует отдельно: начинаем с single-file или flat module и вводим группы `ui`, `model`, `lib`, `api` либо подмодули только после появления реальной сложности.

**Рассмотренные варианты**

- Начать без явной архитектурной формы и раскладывать код по мере появления экранов.
- Использовать ED medium со слоями `app`, `features`, `services` и `shared` и запретом прямых зависимостей между фичами.
- Использовать ED small для текущего масштаба приложения и команды, сохранив возможность перейти к ED medium при росте сложности.

**Последствия**

Это задает будущим разработчикам и агентам проверяемые границы для композиции приложения, фич и shared-инфраструктуры без преждевременного слоя `services`. Код страниц принадлежит соответствующим фичам, а композиция нескольких фич — слою `app`. Цена решения — нужно поддерживать lint-ограничения и public API модулей, а также не превращать ED в папочную церемонию: новые модули начинают с простой структуры и усложняют только при реальной необходимости.
